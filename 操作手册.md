# A股类缠论二买系统 V1 操作手册（方案 D）

## 1. 当前系统定位

本系统当前阶段的目标不是直接实盘交易，而是先把数据链路跑稳：

```text
真实股票基础表
→ 新浪实时批量校验
→ 历史K分阶段缓存
→ 股票池过滤
→ 二买扫描
→ 分级推送
```

方案 D 的核心原则：

```text
数据源不可用就停止；
不再扫无意义的假代码；
实时行情、历史K、股票基础表分开管理；
盘中数据不能污染完整日线。
```

---

## 2. 数据源分工

| 数据类型 | 主源 | 备用 / 说明 |
|---|---|---|
| 股票基础表 | AkShare `stock_info_a_code_name()` | 本地 `stock_basic` 缓存，默认 7 天刷新一次 |
| 实时行情 | 新浪 `hq.sinajs.cn/list=` | 单线程，400只/批，10秒间隔 |
| 历史日K | 腾讯 | efinance / 新浪历史K兜底 |
| 本地缓存 | DuckDB | `data/stock_data.duckdb` |
| 原始数据 | DuckDB raw 表 + 日志 | 方便排查接口字段变化 |

---

## 3. 股票代码范围

系统只保留以下 A 股代码段：

```text
上交所主板：600xxx、601xxx、603xxx、605xxx
科创板：688xxx
深交所主板：000xxx、001xxx、002xxx、003xxx
创业板：300xxx、301xxx
北交所：920xxx
```

不再默认使用 430、830-873、689 等旧号段。

---

## 4. 新浪实时接口规则

新浪实时接口必须严格按以下格式：

```bash
curl "http://hq.sinajs.cn/list=sh000001" \
-H "Referer: http://finance.sina.com.cn" \
-H "User-Agent: Mozilla/5.0"
```

个股示例：

```bash
curl "http://hq.sinajs.cn/list=sz000001" \
-H "Referer: http://finance.sina.com.cn" \
-H "User-Agent: Mozilla/5.0"
```

批量请求：

```text
http://hq.sinajs.cn/list=sh600519,sz000001,sz300750
```

系统默认：

```text
400只/批
单线程
批次间隔 10 秒
请求头必须带 Referer + User-Agent
失败后指数退避重试：1秒 → 2秒 → 5秒
```

新浪实时接口用途：

```text
1. 校验股票代码是否有效
2. 获取盘中当前价
3. 尾盘确认
4. 15:05 后补当天日线
```

注意：新浪实时接口只返回一条最新数据，不返回多条历史K线。

## 7. 构建股票基础表

默认使用缓存，缓存 7 天内有效：

```bash
python run_scan.py --build-basic
```

强制刷新：

```bash
python run_scan.py --build-basic --refresh-basic
```

如果 AkShare 失败但已有本地 `stock_basic` 缓存，系统会使用旧缓存并警告；如果 AkShare 失败且没有缓存，系统停止。

## 9. 构建历史K缓存

```bash
python run_scan.py --build-daily-cache
```

默认线程数为 1，避免免费接口被拒绝。

调试时限制数量：

```bash
python run_scan.py --build-daily-cache --daily-limit 100 --daily-workers 1
```

正式时可以小幅提高：

```bash
python run_scan.py --build-daily-cache --daily-workers 2
```

历史K来源优先级：

```text
DuckDB缓存
→ 腾讯历史K
→ efinance历史K
→ 新浪历史K兜底
```

首次构建会拉最近约 520 根日K；后续通过 `code + date` upsert 合并。

## 16. 常见问题

### Q1：还会不会出现“证券代码可能有误”？

正常流程不会大量出现，因为系统不再扫静态号段，而是使用 AkShare 真实股票基础表，再用新浪实时校验。

如果仍然出现，通常是：

```text
1. stock_basic 过期或异常
2. 新浪实时校验未执行
3. 手动传入了无效代码
4. efinance 对历史K返回了无效代码提示
```

### Q2：为什么不再默认 static fallback？

因为 static fallback 会生成大量不存在代码，跑出来的结果没有意义，且耗时很久。

### Q3：为什么新浪实时批量不能并发？

为了避免空响应、429、连接重置和 IP 临时封禁。系统默认单线程，400只/批，间隔10秒。

### Q4：为什么 15点前不补 daily？

15点前的数据是未完成K线，只能写入 `realtime_quote`，不能污染 `stock_daily`。

### Q5：历史K是不是每天都要全量拉？

数据库层是 upsert 合并。即使接口返回最近 520 根，已有 `code + date` 会覆盖/去重，后续不会重复污染。

## 11. 方案 E：历史 K 稳定化

本版本在方案 D 的数据入口基础上，进一步增强历史 K 源稳定性。

### 11.1 历史 K 获取顺序

系统现在按以下顺序处理历史 K：

```text
1. 先查 DuckDB 本地缓存
2. 缓存足够且最近，直接返回，不请求远程
3. 缓存不足或过旧，尝试腾讯历史 K
4. 腾讯失败，尝试新浪历史 K
5. 新浪历史失败，尝试 efinance
6. 远程全部失败但缓存可用，返回缓存
7. 远程全部失败且无缓存，记录失败
```

这样可以避免每次都重新打远程接口，也能避免接口波动时系统完全不可用。

### 11.2 每个历史 K 源的默认限速

```text
腾讯历史 K：1 秒 / 次
新浪历史 K：1 秒 / 次
efinance：2 秒 / 次
```

可以用环境变量调整：

```bash
export TENCENT_HISTORY_INTERVAL=1.0
export SINA_HISTORY_INTERVAL=1.0
export EFINANCE_HISTORY_INTERVAL=2.0
```

### 11.3 失败冷却

如果某个代码在某个数据源上失败，系统会对这个“代码 + 数据源”做冷却，默认 30 分钟内不重复请求。

```bash
export SOURCE_COOLDOWN_SECONDS=1800
```

### 11.4 缓存有效期

历史 K 缓存默认 7 天内视为可用。

```bash
export CACHE_MAX_STALE_DAYS=7
```

### 11.5 历史 K 构建报告

执行：

```bash
python run_scan.py --build-daily-cache --daily-workers 1
```

结束时会输出类似：

```text
历史K构建报告：
  总处理：100
  成功：95
  失败：5
  缓存命中：80
  腾讯成功：10
  新浪历史成功：3
  efinance成功：2
```

重点看：

```text
缓存命中越高，说明系统越稳定、越省接口。
request_or_parse_failed 越低，说明远程源越健康。
```

### 11.6 daily-workers 建议

默认：

```bash
--daily-workers 1
```

最大建议：

```bash
--daily-workers 2
```

如果传入大于 2，系统会强制降到 2，避免免费接口被打爆。

### 11.7 推荐数据准备流程

第一次：

```bash
python run_scan.py --init-db
python run_scan.py --preflight
python run_scan.py --build-basic
python run_scan.py --validate-basic
python run_scan.py --build-daily-cache --daily-workers 1
```

后续每天：

```bash
python run_scan.py --validate-basic
python run_scan.py --patch-daily-from-sina
python run_scan.py --build-daily-cache --daily-workers 1
```

如果已经有足够缓存，`build-daily-cache` 会优先命中缓存，不会每只股票都打远程接口。

# 方案 F 更新：本轮日志隔离 + 股票池/扫描只读缓存

## 1. 本轮日志隔离

每次执行 `run_scan.py` 会生成独立 `run_id`，日志不再混用当天旧记录。

生成文件示例：

```text
logs/scan_20260424_185031.log
logs/rejects_20260424_185031.csv
logs/raw_bad_rows_20260424_185031.jsonl
```

运行结束后的过滤统计只统计本轮 `rejects_<run_id>.csv`，不会再把当天历史失败记录混进去。

## 2. 数据覆盖率检查

查看当前缓存覆盖率：

```bash
python run_scan.py --coverage
```

输出包括：

```text
stock_basic 数量
stock_daily 覆盖股票数量
realtime_quote 覆盖股票数量
日线覆盖率
实时行情覆盖率
```

如果日线覆盖率较低，扫描结果只代表已缓存股票，不代表全市场。

## 3. 股票池构建规则

从方案 F 开始：

```text
build-universe 只读取本地 stock_daily 缓存
不会在股票池构建阶段远程拉历史K
```

如果某只股票没有日线缓存，会被跳过：

```text
reason = no_daily_cache
```

因此构建股票池前，请先运行历史K缓存：

```bash
python run_scan.py --build-daily-cache --daily-workers 1
python run_scan.py --build-universe --workers 2
```

测试少量股票：

```bash
python run_scan.py --build-daily-cache --daily-limit 100 --daily-workers 1
python run_scan.py --build-universe --limit 100 --workers 1
```

## 4. 扫描阶段默认只读缓存

`observe / tail_confirm / after_close` 默认只读本地 `stock_daily` 缓存，不远程拉历史K。

```bash
python run_scan.py --mode observe
python run_scan.py --mode tail_confirm
python run_scan.py --mode after_close
```

如果你确实想在扫描时允许远程补拉历史K，可以显式加：

```bash
python run_scan.py --mode observe --allow-remote-in-scan
```

但正式使用不建议开启。扫描阶段应该尽量快、稳定、可复现。

## 5. 推荐流程

首次或扩容缓存：

```bash
python run_scan.py --preflight
python run_scan.py --build-basic
python run_scan.py --validate-basic
python run_scan.py --build-daily-cache --daily-workers 1
python run_scan.py --build-universe --workers 2
```

日常运行：

```bash
python run_scan.py --validate-basic
python run_scan.py --patch-daily-from-sina
python run_scan.py --build-universe --workers 2
python run_scan.py --mode observe
```

尾盘：

```bash

```

收盘后：

```bash
python run_scan.py --patch-daily-from-sina
python run_scan.py --build-universe --workers 2
python run_scan.py --mode after_close
```
